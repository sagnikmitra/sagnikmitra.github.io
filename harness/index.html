§ 01Your multi-agent setup is not architecture.

Hey there. So somewhere in the last year, "multi-agent system" became the thing everyone has and almost nobody can explain. You know the setup — a planner agent, a researcher agent, a coder agent, a reviewer agent, and one executor agent that quietly does the work of all four while the others take credit (we've all built this, no judgment).

Here's the uncomfortable part: that's not a system. That's distributed prompt choreography in a trench coat. Five language models taking turns is not architecture, the same way five people in a WhatsApp group is not a company.

And before anyone @'s me — I'm not saying agents are bad. I'm saying the agents are the easy part. They're replaceable cognitive workers. The hard part — the part that decides whether your thing survives contact with production — is the harness: the boring control layer wrapped around the agents that decides who's allowed to do what, who owns the truth, and what happens when everything catches fire at 2 AM (and it will, my god, it will).

NIST frames AI risk as a lifecycle problem — design, development, use, evaluation — not a "which model did you pick" problem [1]. That matters here, because agentic systems act over time, across tools, and across changing state. Three things a single clever prompt has never had to deal with.

The one line that matters Agents may be autonomous inside their lane. The system must never be uncontrolled as a whole. Print it, frame it, tattoo it on your CI pipeline. Everything below is just engineering that sentence into something you can deploy.

§ 02A few definitions, so we're on the same page.

Agent — a bounded actor that observes context, reasons over a task, produces artifacts, and maybe calls a tool. Bounded. Write that down.

Harness — the runtime control envelope around the agents: policy, permissions, budgets, schemas, memory, verification, observability, recovery. The unsexy stuff. The stuff that works.

Orchestration — how you decompose goals, route work, schedule it, merge it, and move state forward without losing your mind.

Notice the harness has nothing to do with how smart your model is. That's the point. You make probabilistic agents behave by putting them inside deterministic boundaries.

§ 03The five planes (where most systems quietly fail).

Serious agent systems separate five planes. Weak systems mash all five inside one giant system prompt and then act surprised.

  1. Control plane — owns authority. Delegation, cancellation, escalation, approval, recovery. Who's the boss.
  2. Data plane — moves events, messages, retrieved docs, tool outputs, telemetry between services. The plumbing.
  3. State plane — owns truth. Task status, locks, memory, customer records, audit trails. The single source of "what actually happened."
  4. Execution plane — does the work. LLM calls, code edits, browser automation, DB queries, shell commands.
  5. Assurance plane — constrains risk. Policy, verification, sandboxing, rollback, human approval. OWASP straight-up recommends least privilege, validated output formats, human approval for high-risk actions, and adversarial testing for prompt injection [4]. Non-negotiable.

If your "multi-agent system" has all five of these living inside the prompt of one supervisor agent — and be honest — that's not five planes. That's one plane wearing five hats and hoping.

§ 04Agents should have contracts, not personalities.

This is the hill I'll gently die on.

Stop defining agents with motivational prompts. "You are a brilliant, meticulous senior engineer who cares deeply about quality" — that's not an engineering spec, that's a horoscope. Define them with contracts:

Allowed tools, forbidden tools, input schema, output schema, budget limits, retry limits, escalation thresholds. The narrower the boundary, the easier the 2 AM debugging session. A verifier that literally cannot call payment_execute can't accidentally pay someone. Revolutionary, I know.

§ 05Orchestration patterns (and when to actually use them).

No single pattern wins everything. Mature systems mix them based on risk, latency, and how much you'll cry if something goes wrong.

  • Hierarchical Supervisor — one boss decomposes the mission, hands bounded tasks to specialists, a verifier checks the work. Safest default for SaaS, fintech, legal-tech, healthcare. Strength: clean accountability, easy to audit. Weakness: the supervisor becomes a bottleneck.
  • Blackboard — agents post to a shared structured workspace. Great for research synthesis, diagnostics, product discovery. But every entry must carry evidence, author, confidence, and verification status — otherwise your blackboard rots into a noisy group chat. (We've seen that group chat. We don't want that group chat.)
  • Event-Driven Mesh — an event bus decouples producers and consumers. Scales beautifully, causal reasoning gets painful unless traces are mandatory. OpenTelemetry's whole model — traces, metrics, logs, baggage [2] — maps directly onto agent runtime instrumentation, so use it.
  • Auction / Market — agents bid on tasks by capability, cost, latency, risk. Useful for robot fleets and heterogeneous workers [6]. One rule: don't allocate on speed or cost alone. That's how you get fast, cheap, unsafe shortcuts.
  • Hybrid — what every serious system actually becomes. Of course it's a hybrid. It's always a hybrid.
What you needPattern that fits
Mission controlHierarchical supervisor
Dynamic allocationAuction / market
Shared reasoningBlackboard
Tool propagationEvent-driven mesh
Safety approvalVerifier + human gate
MemoryCanonical state + event log
 Table 1. Pick the pattern by the job, not by the demo.

§ 06The harness layer, up close.

The harness is what separates "industry-grade autonomous system" from "distributed hallucination machine." A real one includes identity & access control, tool whitelists, data egress controls, output validation, risk scoring, budget ceilings, approval gates, sandboxed execution, rollback, and immutable audit logs. (LangGraph leans hard into durable execution, persistence, human-in-the-loop, and runtime visibility for exactly this reason [3].)

§ 07Verification: the executor does not grade its own homework.

This is the single most skipped step, and it's the one that saves you. Verification must be independent and risk-weighted — the agent that did the work is the last one who should confirm it's correct.

ActionWhat must verify it
Read-only retrievalProvenance + schema
Database writeSchema + permission + state version
Code executionSandbox + tests + static analysis
Payment or tradePolicy + risk + human approval
Customer messageFactuality + tone + compliance
Production deployCI + security + rollback plan
 Table 2. Risk-weighted verification matrix.

A read costs you a schema check. A payment costs you a human. Price the verification to match the blast radius. Simple.

§ 08State, memory, and the failure nobody plans for.

Multi-agent systems fall apart the moment "state" gets vague. Use three layers — local working memory, a shared mission workspace, and one canonical state that is the truth. And agents should never mutate canonical state directly. They propose transitions; the harness verifies and commits. (Yes, like a pull request. Agents open PRs against reality. I think that's beautiful.)

And please — design the unhappy path before you celebrate the happy one:

FailureRecovery
Hallucinated outputVerifier rejects → repair or reassign
Tool timeoutRetry with backoff → fallback tool
Infinite loopRetry-budget kill switch
Prompt injectionIsolate content → block the tool call
Compromised agentRevoke its identity and tools
 Table 3. Common failure modes and their recovery paths.

Every mission, task, tool call, and decision gets a trace_id. Once agents go async, explainability becomes a distributed-systems problem, not a vibes problem.

§ 09The maturity ladder (your "L4" is an L2 in a nice jacket).

Most teams claim L4 while running L2 with extra prompts. The difference was never the number of agents. It's governance, verification, state, observability, and recovery.

LevelWhat it actually means
L0Human uses tools manually
L1Assistant suggests actions
L2Agent uses tools under human control (you are probably here)
L3Multiple agents coordinate tasks
L4Harnessed autonomy with verification + state (you think you are here)
L5Adaptive governed autonomy with self-repair
 Table 4. Autonomy maturity levels.

No shame in L2. A lot of shame in shipping L2 while believing it's L4.

§ 10So where does this actually pay off? Agentic coding.

This is where the theory turns into money. The goal of agentic coding is not faster code generation — it's faster product delivery without collapsing quality, security, or user trust. A real SaaS needs auth, tenant isolation, billing, observability, audit logs, rollback, cost control. Screens and endpoints are the easy 20%.

So you give each concern an agent with a contract — Product, UX, Architecture, Backend, Frontend, QA, Security, DevOps, and a Verifier that blocks the release if tests, tenant isolation, or audit logging are missing.

And the one rule that holds the whole thing together:

The rule that holds it together The same agent must never define the requirement, write the code, approve the diff, disable the test, deploy it, and interpret the failure. That's not autonomy. That's one unsupervised intern with root access. Builders create, verifiers challenge, policy decides, CI enforces, humans approve the scary stuff.

MCP helps standardize how coding agents plug into files, databases, and tools [5] — but standard connectivity just makes least privilege and tool isolation matter more, not less.

§ 11The ten rules, if you skipped to the bottom (I see you).

  1. Start hierarchical before you go full swarm.
  2. Define canonical state before agent memory.
  3. Define contracts before prompts.
  4. Add verification before adding agents.
  5. Separate planning, execution, and approval.
  6. Trace every agent and tool call.
  7. Treat all external content as untrusted.
  8. Sandbox code and browser automation.
  9. Human-gate the high-blast-radius actions.
  10. Optimize for recoverability, not demo magic.

§ 12So, the takeaway.

Multi-agent systems aren't powerful because they have many agents. They're powerful when responsibilities are partitioned, autonomy is bounded, authority is explicit, state is coherent, verification is independent, and failures are recoverable. Everything else is a very expensive way to generate confident nonsense at scale.

Final principle The agents were never the hard part. Build the harness. That's the whole difference between an industry-grade autonomous system and a distributed hallucination machine.
